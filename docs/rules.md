# ReXamples Rules

## 1. One lesson, one concept.
Each example teaches one core concept clearly and completely.

## 2. Native first, always.
Use native ReAction, Intuition, BOOPSI, and standard AmigaOS patterns.

## 3. Plain C89 only.
Lessons must stay readable, disciplined, and free of unnecessary language cleverness.

## 4. Every resource has an owner.
Libraries, classes, objects, ports, signals, buffers, and other resources must have explicit ownership and explicit release.

## 5. Cleanup is part of the design.
Initialization, shutdown, and failure rollback are mandatory parts of every lesson.

## 6. Event-driven means explicit.
The event loop, dispatch path, and exit conditions must remain visible and understandable.

## 7. Abstraction may clarify, but never conceal.
Helper code is acceptable only when it makes the lesson easier to understand without hiding essential mechanics.

## 8. Lessons teach correct habits, not shortcuts.
Anything that would be unacceptable in disciplined native code is unacceptable in ReXamples teaching code.

## 9. Naming must teach the native model.
Use names that reflect native ReAction and AmigaOS concepts, not inherited terminology from other GUI systems.

## 10. Every lesson must be trustworthy as reference code.
If a reader copies the structure, they should learn the right way to build disciplined native ReAction software on AmigaOS 3.2+.
